================================================================================
  AUTHENTICATION PROTOCOL (confirmed March 2, 2026)
================================================================================

  The web service uses:
  - WS-Addressing (2004 namespace): REQUIRED for all calls
    Namespace: http://schemas.xmlsoap.org/ws/2004/08/addressing
  - WS-Security UsernameToken: REQUIRED for authentication
    Namespace: http://docs.oasis-open.org/wss/2004/01/oasis-200401-wss-wssecurity-secext-1.0.xsd

  Authentication is NOT HTTP Basic Auth or NTLM.
  Credentials go in the SOAP Header as a WS-Security UsernameToken.

================================================================================
  KNOWN ISSUES (as of March 2, 2026)
================================================================================

  ** SERVER-SIDE BUG: UsernameToken processing is broken **

  The MiSys server crashes with "Index was outside the bounds of the array"
  whenever ANY UsernameToken is submitted - regardless of credentials.

  Diagnostic summary:
    - No Security header     -> "security header not present" (correct behavior)
    - Timestamp only         -> "UsernameToken is expected" (correct behavior)
    - Empty UsernameToken    -> "Invalid security token" (correct behavior)
    - ANY username+password  -> SERVER CRASH: "Index was outside the bounds"
    - Wrong password (test/test) -> SAME CRASH

  This means:
    1. The web service IS running and the WSDL loads correctly
    2. WS-Addressing headers work correctly
    3. WS-Security Timestamp processing works correctly
    4. The UsernameToken validation code has an internal server error
    5. Credentials CANNOT be verified until the server bug is fixed

  DEFINITIVE PROOF (March 2, 2026):
    - Used the EXACT SAME WSE 3.0 DLL (Microsoft.Web.Services3) from GAC
    - Created proxy inheriting from WebServicesClientProtocol (same as MiSys client)
    - Added UsernameToken via RequestSoapContext.Security.Tokens.Add()
    - Tested ALL password options: SendPlainText, SendHashed, SendNone
    - Tested ALL known users: ADMIN/password (sample), HARON, AlhakimiH
    - ALL combinations crash with "Index was outside the bounds of the array"
    - The MiSys default ADMIN/password account also crashes

  Likely cause:
    - MiSys WSE 3.0 CustomUsernameTokenManager has a broken dependency
    - The MiSys SQL Server instance (MSSQL12.MISYSSBMSE) may be down
    - The authentication database/store the server references may be down
    - WSE configuration on the server may be corrupted

  Resolution:
    - Check if the MiSys SQL Server instance (MISYSSBMSE) is running on 192.168.1.11
    - Check IIS Application Pool for sbmwebservice64 is running
    - Check IIS logs and Windows Event Viewer for full .NET stack traces
    - May need to restart the MiSys web service / IIS / SQL Server
