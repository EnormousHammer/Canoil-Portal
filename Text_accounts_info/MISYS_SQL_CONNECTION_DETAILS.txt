   5. HOW WE FOUND THE SQL CREDENTIALS (step by step)
============================================================

  If you ever need to re-derive these credentials:

  1. Found SQL Server instance name from Windows registry:
     Path: HKLM:\SOFTWARE\WOW6432Node\MISys\SBM\6.4
     Key "ServerBackupDir" contained:
       C:\Program Files (x86)\Microsoft SQL Server\MSSQL12.MISYSSBMSE\MSSQL\backup\
     This revealed: instance = MISYSSBMSE, SQL 2014 (MSSQL12)

  2. Confirmed port 1433 was open on 192.168.1.11 via TCP test

  3. SQL Browser (UDP 1434) was NOT running, so named instance
     lookup times out. But the instance listens on port 1433.

  4. Found MiSysKeyMgr.Client.dll.config in:
     C:\Program Files (x86)\MISys\MISys SBM\
       MISys SBM 6.4 Client\MiSysKeyMgr.Client.dll.config

  5. That config had DPAPI-encrypted user settings
     (DataProtectionConfigurationProvider / CipherData)

  6. Used .NET ProtectedData.Unprotect() with
     DataProtectionScope.CurrentUser to decrypt

  7. Inside the decrypted XML, found Base64-encoded values:
       SA_U    = c2E=                -> "sa"
       SA_P    = TUlTeXNfU0JNMQ==   -> "MISys_SBM1"
       SA_OldP = TUlTeXMx           -> "MISys1"

  8. Connected with sa / MISys_SBM1 -- full access confirmed.

  IMPORTANT: The DPAPI decrypt only works when run as the
  SAME WINDOWS USER who was logged in when MiSys was
  installed/configured. If the machine is reimaged or a
  different user runs the script, DPAPI will fail. But the
  SQL credentials themselves still work regardless.

  PowerShell script to repeat the DPAPI decrypt:

    $configPath = "C:\Program Files (x86)\MISys\MISys SBM\MISys SBM 6.4 Client\MiSysKeyMgr.Client.dll.config"
    [xml]$xml = Get-Content $configPath -Raw
    $cipher = $xml.configuration.userSettings.'MiSysKeyMgr.Client.Properties.UserSettings'.EncryptedData.CipherData.CipherValue.Trim() -replace '\s+',''
    Add-Type -AssemblyName System.Security
    $bytes = [Convert]::FromBase64String($cipher)
    $dec = [System.Security.Cryptography.ProtectedData]::Unprotect($bytes, $null, [System.Security.Cryptography.DataProtectionScope]::CurrentUser)
    [System.Text.Encoding]::UTF8.GetString($dec)

============================================================
   11. THINGS THAT DID NOT WORK (avoid repeating)
============================================================

  WEB SERVICE AUTHENTICATION:
    We spent significant time trying to authenticate via the
    SOAP web service at miwebservice.asmx. Results:
    - GetPwd method worked via MISourceComponent wrapper
    - Most other methods returned MISYSer700 (auth failure)
    - Some methods returned ERRCODE 754 (licensing error
      from MiSysKeyMgr.Client.MiSysKeyMgrClient)
    - The MiSysKeyMgr licensing system cannot initialize
      outside the full MiSys desktop client runtime
    - Direct SOAP calls with manually constructed tokens
      returned 500 errors

    full MiSys client. Direct SQL is simpler and gives MORE
    access (all tables, all columns).
